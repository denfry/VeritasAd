## Контекст и цель

M1 построил лёгкий декаплингованный пакет `research/corpus/`, который из списка URL (VK/RuTube/YouTube) майнит регуляторную маркировку (ERID/`#реклама`/ИНН/промокоды) и выдаёт версионированный JSONL-манифест **weak-меток** (`disclosed_ad` vs `unlabeled`) — фундамент дистанционного надзора. Живой smoke-run подтверждён на реальных данных: реальные ERID-токены извлекаются из описаний RuTube, видео корректно метятся `disclosed_ad`.

**M2** — следующий программный этап: построить **кэшируемый, версионируемый мультимодальный набор признаков** по видео из корпуса, чтобы потом обучать модель детекции (M4) и честно её оценивать (M5). Это ядро научного метода: именно мультимодальное слияние под слабым надзором (ERID как distant-supervision) + темпоральная локализация рекламы — вероятная ось научной новизны.

В отличие от продакшн-пайплайна в `backend/` (реальное время, Celery/DB/Redis), M2 — **офлайн, воспроизводимый**: тяжёлые модели гоняются один раз, результат складывается в портируемый кэш.

## Секция 2 — Пакет `veritas-features` и стратегия рефакторинга бэкенда *(согласовано)*

**Что выносим (и только это):** низкоуровневые *примитивы* извлечения на базе моделей. Высокоуровневые продакшн-эвристики бэкенда (бренд-детекция по CLIP-промптам, агрегация, ad-keyword-скоринг) **остаются в бэкенде** и зовут эти примитивы. Это сужает поверхность рефакторинга и риск.

**Поверхность `veritas_features` (чистые функции, без DB/Celery/Redis/config-сайд-эффектов):**
- `audio.py`: `extract_audio_wav(video) → wav` · `transcribe(wav, model) → Transcript` · `mfcc_windows(wav, win, overlap) → list[MfccWindow]`
- `visual.py`: `sample_frames(video, interval) → list[Frame(ts, image)]` · `clip_embed(frames, model) → list[FrameEmbedding(ts, vec)]` (**сырые** эмбеддинги, не бренд-детекция)
- `ocr.py`: `ocr_frames(frames) → list[OcrSpan(ts, text, conf)]` (препроцессинг CLAHE/denoise/Otsu + tesseract→easyocr fallback)
- `models.py`: ленивая загрузка CLIP/Whisper, авто-детект CUDA, размер модели через параметр/env (без зависимости от backend-config)
- `types.py`: dataclasses-выходы с таймкодами

**Почему сырые эмбеддинги, а не бренд-детекция:** для научной новизны M4-модель учится на *выученных* представлениях (CLIP/MFCC-векторы + ASR/OCR-текст), а не на ручных эвристиках. Инженерные бренд-эвристики — продакшн-часть бэкенда.

**Текст/раскрытие — НЕ в тяжёлый пакет.** Лёгкая stdlib-логика дисклоуз-майнинга уже в `corpus/erid_miner`. M2 переиспользует её напрямую.

**Безопасный рефакторинг бэкенда (порядок Плана A):**
1. **Базлайн:** тесты бэкенда зелёные. Если покрытие этих функций тонкое — сперва добавить характеризующие (regression) тесты на текущее поведение.
2. Создать `veritas-features` + перенести примитивы **по одной модальности** (audio → visual → ocr), TDD на чистых трансформах.
3. После каждой модальности: `backend/app/services/{audio_analyzer,video_processor,brand_ocr}.py` импортируют примитив и становятся тонкой обёрткой; **тесты бэкенда → остаются зелёными** (поведение не меняется).
4. Добавить `veritas-features` как path-dependency в `backend` (pyproject/requirements).

**Якоря в коде бэкенда (из разведки) для переноса:**
- Audio: `backend/app/services/audio_analyzer.py` — extract_audio (74–117), transcribe (119–161), mfcc_windows (194–231).
- Visual: `backend/app/services/video_processor.py` — frame extraction (624–693), `_detect_with_clip` (920–1033, низкоуровневая часть).
- OCR: `backend/app/services/brand_ocr.py` — preprocess (133–155), extraction (173–209).
- Загрузка моделей: `backend/app/services/model_manager.py` (singleton CLIP/Whisper).

**Явно вне области:** Celery/DB/Redis-оркестрация, прогресс-колбэки, облачные детекторы (Azure/AWS, сеть), LLM.

## Секция 3 — M2 research-пайплайн: кэш, схема, резюмируемость *(проектируется)*

Предварительный замысел (требует доработки и согласования):
- **`media.py`** — скачивание видео через yt-dlp в локальный медиа-кэш, извлечение аудио, политика очистки (хранить/удалять — конфигурируемо), резюм по уже скачанному.
- **`extract.py`** — оркестратор: на каждое видео корпуса зовёт `veritas_features` по 4 модальностям, собирает `FeatureRecord` (нативные сегменты + таймкоды).
- **Формат кэша** (предв.) — на видео: `clip.npz`/`mfcc.npz` (массивы + таймкоды), `asr.jsonl`/`ocr.jsonl` (сегменты), `meta.json` (video_id, `FEATURE_VERSION`, версии/размеры моделей, провенанс, время). Глобальный `features.manifest.jsonl` — как манифест корпуса.
- **`resample.py`** — дешёвая нарезка нативных сегментов на фиксированную сетку для обучения (без переизвлечения).
- **`cli.py`** — резюмируемый прогон: пропуск видео, уже извлечённых на текущей `FEATURE_VERSION`.

Открытые вопросы для доработки: точный формат кэша (npz vs parquet vs zarr), политика хранения скачанных видео, схема `FeatureRecord`, выравнивание таймкодов между модальностями.

## Секция 4 — Тестирование и воспроизводимость *(проектируется)*

Предварительный замысел:
- **veritas-features:** unit-тесты чистых трансформов; модель-зависимые функции — на крошечных фикстурах (1-сек wav, один кадр) или как integration-тесты за флагом.
- **M2:** чистые модули (схема, store I/O, resample, манифест) — TDD без тяжёлых моделей; оркестратор — с **фейковым core** (паттерн `FakeFetcher` из M1); один маленький живой smoke-run.
- **Воспроизводимость:** константа `FEATURE_VERSION` (bump → переизвлечение); пиннинг id/размеров моделей в конфиге и запись в `meta.json`; сиды; портируемые артефакты кэша (извлечение на стороннем GPU → копирование).
